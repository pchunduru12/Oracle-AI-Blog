01 — What it is

What OpenClaw actually is

OpenClaw is not a SaaS product, not an enterprise orchestration platform, and not something you buy a licence for. It is an open-source, MIT-licensed, local-first AI agent framework that you install on your own machine and run as a persistent background daemon.

Created by Peter Steinberger and launched in January 2026, it reached 100,000 GitHub stars in under two weeks — one of the fastest-growing open-source repos in GitHub history. The reason is simple: it solves the gap between "I can talk to an AI" and "the AI acts on my behalf while I'm not watching."

The key distinction for Oracle delivery: OpenClaw is the runtime that executes your agent logic. The Oracle AI Codex — your object naming conventions, validation rules, business logic, deployment standards — becomes the agent's memory. OpenClaw reads that memory on every run and uses it to drive tool calls against Oracle EBS or Fusion Cloud.

Three things that make it different

🏠

Local-first

The gateway, memory, and tools run on hardware you control. Memory is stored as plain Markdown files on disk — inspectable, version-controllable, auditable.

⏰

Autonomous

A heartbeat daemon wakes the agent on a schedule without being prompted. Your AP validation checks run at 17:00 whether or not anyone triggers them manually.

🔌

Model-agnostic

Connects to Claude, GPT-4o, Gemini, or fully local Ollama models. For Oracle financial work, Claude Sonnet is recommended for its structured-output accuracy.

How the runtime works

When you run openclaw gateway, a single long-lived Node.js process starts — called the Gateway. That one process handles everything: channel connections (Teams, Slack, Telegram, etc.), session management, the agent loop, model calls, tool execution, and memory persistence. There is no separate service to manage.

SOUL.md, AGENTS.md, TOOLS.md, MEMORY.md — Oracle rules, object names, validation logic

⚙️ Gateway :18789

Long-running Node.js daemon. Loads workspace, manages sessions, runs tool calls, fires heartbeats

🤖 LLM + Tools

On each run — whether triggered by a cron heartbeat, an incoming Teams message, or a webhook — the agent runtime assembles context from all workspace files, calls the model, executes any tool calls the model requests, feeds results back, and repeats until done. Every step runs in the Gateway process on your machine.

02 — Installation

Download and install OpenClaw

You do not need to clone the repository to use OpenClaw. Install it globally via npm and run the onboarding wizard. The whole setup takes under 30 minutes.

Prerequisites: Node.js 22 or later (required — not 18 or 20). An API key from at least one LLM provider. A machine that stays running: a Mac, a Linux server, a VPS, or Oracle Cloud Free Tier (4 ARM cores, 24 GB RAM, permanently free — ideal for Oracle teams).
01
Install the CLI globally

This installs the openclaw CLI and the Gateway daemon. No cloning required.

npm install -g openclaw@latest
02
Run the onboarding wizard

Creates ~/.openclaw/workspace/, seeds all template files (SOUL.md, AGENTS.md, TOOLS.md, MEMORY.md), and walks through LLM provider setup and channel connection. The --install-daemon flag registers the Gateway as a systemd/launchd service so it survives reboots.

openclaw onboard --install-daemon
03
Add your Anthropic API key

Credentials live in the OpenClaw config, never in workspace markdown files. This is a hard rule for Oracle environments — no credentials in the Codex repo.

openclaw config set providers.anthropic.apiKey "sk-ant-..."
04
Set your model and spending limit

Claude Sonnet is recommended for Oracle financial work. Set a hard monthly spending cap — critical for an always-on agent that could be triggered unexpectedly.

openclaw config set agents.defaults.model "anthropic/claude-sonnet-4-5"
05
Create the Oracle agent workspace

Keep the Oracle close agent in its own isolated workspace so its memory and tool set stay separate from any other agents on the same machine.

openclaw agents add oracle-close --workspace ~/.openclaw/workspace-oracle
06
Connect Microsoft Teams

Teams is the most common notification channel in Oracle delivery. The close agent posts validation reports, escalations, and reconciliation packs directly to a Teams channel.

openclaw channels connect teams
07
Start the Gateway and verify

The doctor command surfaces misconfigured channels, missing credentials, and workspace file issues before your first live run.

openclaw gateway start && openclaw doctor
08
Register the close schedule

Cron tasks run inside OpenClaw's scheduler. This fires pre-close validation at 17:00 on the last five days of each month, with no external cron job needed.

openclaw cron add --agent oracle-close --schedule "0 17 27-31 * *" --task "Run AP pre-close validation for current period. Post results to Teams oracle-finance channel."
03 — Workspace files

The workspace files — Codex as agent memory

Every OpenClaw agent is defined by a set of Markdown files in its workspace directory. These are read on every run to reconstruct the agent's identity, rules, memory, and available tools. This is the direct bridge: your Oracle AI Codex lives here.

The SOUL.md for an Oracle close agent

~/.openclaw/workspace-oracle/SOUL.md
# Oracle Financial Close Agent — SOUL

## Identity
You are an Oracle financial close agent for AP and AR period-end
sub-ledger close on Oracle EBS 12.2 and Oracle Fusion Cloud.
You are an execution agent, not a chatbot. Your job is to run
pre-close validation, escalate exceptions, coordinate the close
sequence, and produce reconciliation evidence.

## Hard constraints — never violate these
- NEVER release AP holds automatically
- NEVER submit a period close without controller sign-off
- NEVER proceed past a gate when there are unresolved exceptions
- NEVER invent business rules — escalate if the Codex is silent
- NEVER run INSERT, UPDATE, or DELETE against any database
- NEVER run DML against PROD — only UAT unless explicitly told otherwise

## On uncertainty
If a check result is ambiguous, stop and escalate. Do not proceed
on assumption. A wrong close is far worse than a delayed one.

## Output style
Produce structured JSON or Markdown that humans can act on directly.
Every escalation must include: the failed check, the record count,
the named owner, and a specific resolution path.
Version-control your workspace as a private Git repo. Every time the controller adds a new exception routing rule, it is a commit. Your Codex accumulates institutional knowledge that survives team changes, and every audit question about "what rules was the agent following in March" has a precise, timestamped answer.
04 — Oracle APIs

Oracle APIs — how the tools connect

OpenClaw's tools are ordinary code — Python scripts, Node.js modules, or shell commands the agent calls by name. The Oracle connection lives inside those tools. Here are the specific APIs each tool wrapper calls.

Oracle Fusion Cloud — two REST layers

Fusion Cloud exposes two separate REST API surfaces for the work a close agent needs. Understanding which one to use for which task is the most common stumbling block.

TaskAPI endpointNotes
Query AP invoices on hold /fscmRestApi/resources/11.13.18.05/invoices fscmRestApi Use query parameter ?q=holdReasonCode IS NOT NULL. Read-only GET.
Get accounting period status /fscmRestApi/resources/11.13.18.05/accountingPeriods fscmRestApi Returns period name, status (Open/Closed/Never Opened), application ID.
Submit ESS job (sweep, transfer to GL) /ess/rest/scheduler/v1/requests Scheduler API Introduced in 23B. Preferred over the older erpintegrations endpoint for new implementations. POST with jobDefinitionName, parameters.
Submit ESS job (legacy / existing integrations) /fscmRestApi/resources/11.13.18.05/erpintegrations erpintegrations Older path still widely used. POST with OperationName: submitESSJobRequest, JobPackageName, JobDefName.
Poll ESS job status /ess/rest/scheduler/v1/requests/{requestId} Scheduler API Returns status: WAIT / RUNNING / SUCCEEDED / FAILED / WARNING. Poll every 60 seconds.
Query AR receipts (unposted) /fscmRestApi/resources/11.13.18.05/receivablesReceipts fscmRestApi Filter with ?q=status=UNPOSTED. Returns receipt number, bank account, amount.
Register ESS callback / webhook /ess/rest/scheduler/v1/requests with callbackURL Scheduler API Register a callback URL so Oracle pushes completion status instead of the agent polling.
Use the Scheduler REST API for new work. Oracle introduced /ess/rest/scheduler/v1/requests in version 23B specifically to support automation, orchestration, and integration of ESS jobs. It supports submit, query, cancel, hold, and callback registration — all the operations a close agent needs. The older erpintegrations endpoint still works but is not being enhanced.

Submitting the AP Sweep via Scheduler REST API

Python · tools/submit_payables_sweep.py
import os, time, requests

def submit_payables_sweep(period: str, operating_unit: str) -> dict:
    """
    Submits the Payables Accounting Periods Close ESS job
    via Oracle Fusion Scheduler REST API (23B+).
    Requires gate_approval_confirmed = True before calling.
    Returns: job_request_id, status
    """
    base = os.environ["FUSION_BASE_URL"]
    auth = (os.environ["FUSION_USER"], os.environ["FUSION_PASS"])

    # Submit the ESS job
    payload = {
        "jobDefinitionName": "APXPCLOSE",       # AP Period Close
        "jobDefinitionPackage": "/oracle/apps/ess/financials/payables/accounting",
        "parameters": [period, operating_unit],
        "callbackURL": os.environ.get("OPENCLAW_WEBHOOK_URL")  # push on complete
    }
    r = requests.post(
        f"{base}/ess/rest/scheduler/v1/requests",
        json=payload, auth=auth,
        headers={"Content-Type": "application/json"}
    )
    r.raise_for_status()
    job_id = r.json()["requestId"]

    # Poll until terminal state (or timeout after 90 min)
    deadline = time.time() + 5400
    while time.time() < deadline:
        status_r = requests.get(
            f"{base}/ess/rest/scheduler/v1/requests/{job_id}", auth=auth
        )
        status = status_r.json().get("requestStatus")
        if status in ("SUCCEEDED", "FAILED", "WARNING"):
            return {"job_id": job_id, "status": status, "period": period}
        time.sleep(60)

    return {"job_id": job_id, "status": "TIMEOUT", "period": period}

Oracle EBS 12.2 — three connection options

EBS does not have a native REST layer as comprehensive as Fusion Cloud. Oracle delivery teams connect OpenClaw tools to EBS via three approaches — choose based on your environment's existing infrastructure:

🔗

ORDS on EBS

Oracle REST Data Services can be deployed on EBS 12.2 to expose custom PL/SQL procedures and views as REST endpoints. The agent's tools call these endpoints — no direct DB connection required from the agent machine.

🐍

python-oracledb

For read-only validation queries, a Python tool wrapper uses python-oracledb against the EBS read replica. The agent calls the function by name; the function runs approved SQL and returns normalised JSON. No raw SQL in the agent.

☁️

Oracle Integration Cloud

For teams already using OIC for EBS integrations, the agent calls OIC REST endpoints and OIC handles the EBS adapter connection. No direct DB access needed — the agent only knows the OIC endpoint URL.

Example: read-only AP holds query for EBS 12.2

Python · tools/check_open_holds_ebs.py
import os, oracledb

def check_open_holds(period: str, operating_unit: str) -> dict:
    """
    Read-only. Queries AP_HOLDS_ALL for unreleased holds in period.
    Never modifies data. Connects to read replica only.
    """
    conn = oracledb.connect(
        user=os.environ["EBS_RO_USER"],
        password=os.environ["EBS_RO_PASS"],
        dsn=os.environ["EBS_RO_DSN"]
    )
    cur = conn.cursor()
    cur.execute("""
        SELECT h.hold_lookup_code,
               COUNT(*)                 AS hold_count,
               SUM(i.invoice_amount)    AS total_held
        FROM   ap_holds_all h
        JOIN   ap_invoices_all i  ON i.invoice_id = h.invoice_id
        JOIN   hr_operating_units ou ON ou.organization_id = i.org_id
        WHERE  h.release_lookup_code IS NULL
        AND    ou.name  = :ou
        AND    TRUNC(i.invoice_date)
                 BETWEEN (SELECT start_date FROM gl_periods
                          WHERE  period_name    = :period
                          AND    application_id = 200)
                     AND (SELECT end_date   FROM gl_periods
                          WHERE  period_name    = :period
                          AND    application_id = 200)
        GROUP BY h.hold_lookup_code
        ORDER BY hold_count DESC
    """, {"ou": operating_unit, "period": period})

    rows = cur.fetchall()
    conn.close()

    total = sum(r[1] for r in rows)
    return {
        "check":   "open_holds",
        "period":  period,
        "status":  "CLEAR" if total == 0 else "EXCEPTIONS_FOUND",
        "total":   total,
        "by_code": [{
            "hold_code": r[0],
            "count":     r[1],
            "amount":    float(r[2] or 0)
        } for r in rows]
    }
05 — Close agent

The AP sub-ledger close agent — end to end

With OpenClaw running, the Codex loaded as workspace files, and Oracle tool wrappers registered, this is what the complete AP close agent looks like — from the AGENTS.md procedure through to what the controller sees in Teams.

The AGENTS.md close procedure

This is the section of AGENTS.md that the agent reads on every close run. It is written as direct instructions, not config — the agent reads it and follows it step by step. Human gates are written explicitly as STOP instructions.

workspace-oracle/AGENTS.md — AP close procedure
## AP Sub-Ledger Close Procedure

When asked to run AP close, follow this sequence exactly.
Never skip a step. Never proceed past a gate unless explicitly confirmed.

### Step 1 — Pre-close validation (you execute)
Call in this order:
  check_open_holds(period, operating_unit)
  check_unapproved_invoices(period, operating_unit)
  check_unposted_invoices(period, operating_unit)
  check_unmatched_receipts(period, operating_unit)
  check_period_status(period, application_id=200)

Collect all results. Call emit_validation_report(results).
Post the report to Microsoft Teams oracle-finance channel.

### Step 2 — Exception resolution gate (HUMAN REQUIRED — STOP)
Do not proceed. The validation report is now in Teams.
Wait for the AP lead and controller to resolve exceptions
and reply with "gate approved" before continuing.
If no response in 4 hours, re-post and tag by name.

### Step 3 — Confirm clear (you execute)
Re-run all five checks from Step 1.
If any exception still exists, return to Step 2.
If all clear, log confirmation timestamp and continue.

### Step 4 — Submit AP Sweep (you execute)
Call submit_payables_sweep(period, operating_unit).
Log the ESS job request ID.
Poll until SUCCEEDED or FAILED.
If FAILED, post error to Teams and stop — do not continue.

### Step 5 — Transfer to GL (you execute)
Call submit_transfer_to_gl(period, operating_unit).
Confirm SUCCEEDED. Log batch count, line count, total amount.

### Step 6 — AP-to-GL Reconciliation (you execute)
Call run_ap_gl_reconciliation(period).
Post reconciliation pack to Teams.
If variance is ANY non-zero value — stop. Do not proceed.

### Step 7 — Controller sign-off gate (HUMAN REQUIRED — STOP)
Controller must reply "recon approved" before period close.
Even a $0.01 variance requires an explanation comment.

### Step 8 — Period close (you execute)
Call submit_period_close(period, operating_unit).
Confirm status changes to Closed.
Call emit_close_completion_log(all_steps).
Post final confirmation to Teams.

## Gate policy
Never skip a gate under any circumstances.
If uncertain about a step result, escalate — never assume and proceed.

06 — Models & costs

Models, costs, and practical sizing

Daily heartbeat monitoring, queue-depth checks, status pingsClaude Haiku 4.5 or Ollama Llama 3.2 locallyNear zero

Realistic annual cost for an Oracle AP close agent running 12 monthly closes plus daily heartbeats: $40–120/year in LLM API fees. Hosting is free on Oracle Cloud Free Tier. Compare that to the 4–6 hours of senior controller time consumed by the same manual process each period.

Set a hard spending limit. OpenClaw can be triggered by incoming messages, webhooks, and scheduled heartbeats — which means it can accumulate API calls faster than expected in a busy environment. Set providers.anthropic.spendingLimit in openclaw.json before connecting to production Oracle environments. A $20/month cap is a reasonable starting point for a single close agent.
