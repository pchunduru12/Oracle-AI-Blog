01 — The problem

Why Oracle teams struggle to get reliable AI output

Custom PL/SQL packages with incomplete history. A staging table nobody fully owns because the original developer is gone. An AI assistant that writes syntactically valid code, but still gets the architecture wrong — wrong naming, wrong layer, wrong deployment assumptions, wrong EBR handling.

The issue is usually not the AI model itself. The issue is that the repository does not expose enough context for the assistant to be dependable.

The real cost
  • New developers ramp slowly because standards live in people’s heads.
  • AI produces generic Oracle output that violates local conventions.
  • Code review becomes standards policing instead of design review.
  • When a developer leaves, institutional memory leaves too.
The way you structure custom Oracle work determines whether AI becomes a force multiplier or just a faster way to generate drafts you still have to rewrite.
A better term than “just another repo”

The Oracle AI Codex

A structured, version-controlled repository of your custom Oracle extensions and integrations — organized so an AI assistant can understand your architecture, standards, examples, and deployment process quickly.

Not a documentation dump and not a PDF that goes stale after week one. A living, AI-readable codex that helps enforce standards at write time — before review, QA, and production.

02 — How it works

The Oracle AI Codex has two main ingredients. First, a consistent directory structure across projects. Second, an AI context file such as AI_CONTEXT.md, CLAUDE.md, or .cursorrules that carries your rules into every interaction.

03 — The platform split

Oracle EBS and ERP Cloud need different AI context strategies

The codex idea works for both platforms, but the information the AI needs is different. That is the key design point.

Oracle EBS R12.x

The AI can learn from DDL and schema structure

  • Direct database access and custom PL/SQL
  • Custom schemas such as XXCUS, APPS, XXAPX, XXORDS
  • Edition-Based Redefinition matters
  • Deployment is ordered SQL execution
  • DDL files teach the AI your data model directly
Oracle ERP Cloud / Fusion

The AI must learn from payloads, mappings, and promotion rules

  • No direct database access
  • Extensions live in OIC, BIP, VBCS, FBDI, Groovy, OTBI
  • Oracle quarterly updates shape your design choices
  • Deployment is promotion across environments
  • REST payload examples become the equivalent of DDL

The most important difference for AI context

In EBS, the AI can inspect DDL and understand the actual structure of tables, columns, constraints, and relationships. In ERP Cloud, there is no DDL to inspect. That means the context file must include real REST payloads, field mappings, authentication patterns, and release expectations. For Cloud teams, payload examples play the role that DDL plays in EBS.

04 — What changes in practice

Without the codex, the AI usually produces isolated snippets. With it, the AI can generate work that respects your packaging, sequencing, and conventions.

What changes

You stop asking the assistant to “write some code” in the abstract. You start asking it to work inside a system that already knows your architecture, naming, dependencies, and release discipline.

05 — The context file

What belongs in the AI context file

Six sections create the “understands our environment” effect. If one is missing, the assistant becomes weaker in that area.

1

Repository overview

What the repository contains, what modules it covers, and what it explicitly does not contain.

2

Architecture

For EBS, schema ownership and access chain. For Cloud, extension layers and which layer owns business logic.

3

Naming conventions

Often the highest-value section. It influences almost every file, object, parameter, and variable the AI creates.

4

Code standards

Logging, exception handling, package structure, comments, sequencing, and reusable templates.

5

Deployment process

The order in which artifacts are created, compiled, promoted, and validated across environments.

6

Project inventory

Real examples that show the assistant what “good” looks like in your environment.

Example naming excerpt
| Object Type      | Pattern                   | Example                    |
|------------------|---------------------------|----------------------------|
| Custom table     | XXCUS_[MOD]_[NAME]        | XXCUS_HR_EMPLOYEES         |

| IN parameter     | p_[name]                  | p_batch_id                 |

07 — How to start

How to start without boiling the ocean

Do not begin by restructuring everything. Start with one active project, one solid context file, and one or two gold-standard examples. The goal is to prove that the structure improves accuracy before you scale it across the entire portfolio.

For Oracle EBS teams
  1. Write the root context file with schemas, naming, EBR flow, and logging standards.
  2. Structure one active project with XXCUS, APPS, UI/API layers, and patch folders.
  3. Extract and store real DDL so the assistant can read the data model directly.
  4. Create one gold-standard example project before expanding broadly.
For ERP Cloud teams
  1. Write the root context file around payloads, auth patterns, field mappings, and promotion rules.
  2. Structure one integration project with OIC, BIP, FBDI, config, and support artifacts.
  3. Add representative payload examples and mapping documents for the highest-value objects.
  4. Capture a release checklist so promotion discipline becomes part of the codex.
